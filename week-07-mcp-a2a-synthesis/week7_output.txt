============================================================
  MCP OVERVIEW — Simulated Protocol Exchange
============================================================

  Think of MCP like USB for AI agents:
  Any tool that speaks the protocol can plug right in.
  No custom adapters needed.

  We will now simulate a complete MCP session:
  1. Client initializes the connection
  2. Client discovers available tools
  3. Client calls a tool and gets results

============================================================
  STEP 1: Initialize Session
============================================================
  The client introduces itself and learns what the
  server supports (capability negotiation).

────────────────────────────────────────────────────────────
  CLIENT → SERVER  [initialize request]
────────────────────────────────────────────────────────────
{
  "method": "initialize",
  "id": 1,
  "params": {
    "protocolVersion": "2025-03-26",
    "clientInfo": {
      "name": "demo-ai-agent",
      "version": "1.0.0"
    },
    "capabilities": {
      "sampling": {}
    }
  },
  "jsonrpc": "2.0"
}

────────────────────────────────────────────────────────────
  SERVER → CLIENT  [initialize response]
────────────────────────────────────────────────────────────
{
  "id": 1,
  "result": {
    "protocolVersion": "2025-03-26",
    "serverInfo": {
      "name": "demo-tool-server",
      "version": "1.0.0"
    },
    "capabilities": {
      "tools": {
        "listChanged": true
      },
      "resources": {
        "subscribe": false
      },
      "prompts": {
        "listChanged": false
      }
    }
  },
  "jsonrpc": "2.0"
}

============================================================
  STEP 2: Discover Available Tools
============================================================
  The client asks the server what tools it offers.
  Each tool comes with a JSON Schema describing its inputs.
  This schema is what gets passed to the LLM so it knows
  how to format tool calls.

────────────────────────────────────────────────────────────
  CLIENT → SERVER  [tools/list request]
────────────────────────────────────────────────────────────
{
  "method": "tools/list",
  "id": 2,
  "params": {},
  "jsonrpc": "2.0"
}

────────────────────────────────────────────────────────────
  SERVER → CLIENT  [tools/list response]
────────────────────────────────────────────────────────────
{
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "get_weather",
        "description": "Get current weather for a city",
        "inputSchema": {
          "type": "object",
          "properties": {
            "city": {
              "type": "string",
              "description": "City name, e.g. 'London'"
            }
          },
          "required": [
            "city"
          ]
        }
      },
      {
        "name": "web_search",
        "description": "Search the web for a query",
        "inputSchema": {
          "type": "object",
          "properties": {
            "query": {
              "type": "string",
              "description": "Search query"
            }
          },
          "required": [
            "query"
          ]
        }
      }
    ]
  },
  "jsonrpc": "2.0"
}

============================================================
  TRANSPORT TYPES
============================================================

  ┌─ stdio ─────────────────────────────
  │ How:      Client spawns the server as a child process.
    Messages flow through stdin (client→server)
    and stdout (server→client).
  │ Best for: Local tools, CLI integrations, desktop apps
  │ Example:  claude desktop → python mcp_server.py
  │ Pros:     Simple, no network needed, secure (local only)
  │ Cons:     Single client only, no remote access
  └──────────────────────────────────────────────────

  ┌─ SSE (Server-Sent Events) ─────────────────────────────
  │ How:      Server runs as an HTTP server.
    Client sends requests via POST.
    Server streams responses via SSE endpoint.
  │ Best for: Remote servers, shared tool services
  │ Example:  agent app → https://tools.example.com/sse
  │ Pros:     Works over network, multiple clients
  │ Cons:     Two endpoints needed (POST + SSE), being replaced
  └──────────────────────────────────────────────────

  ┌─ Streamable HTTP ─────────────────────────────
  │ How:      Single HTTP endpoint handles everything.
    Requests via POST, responses streamed back.
    Supports bidirectional streaming.
  │ Best for: Production deployments (newest, recommended)
  │ Example:  agent app → https://tools.example.com/mcp
  │ Pros:     Single endpoint, full streaming, stateless option
  │ Cons:     Newest — some SDKs still adding support
  └──────────────────────────────────────────────────

============================================================
  CLIENT vs SERVER ROLES
============================================================

  CLIENT (the AI application)          SERVER (the tool provider)
  ─────────────────────────            ──────────────────────────
  • Initiates the connection           • Waits for connections
  • Sends initialize request           • Responds with capabilities
  • Asks for tool/resource lists       • Provides tool/resource lists
  • Decides WHEN to call a tool        • Executes tool calls
  • Passes results to the LLM          • Returns structured results
  • Manages the conversation           • Stateless between calls (usually)
  • One client can connect to          • One server can serve
    MULTIPLE servers                     MULTIPLE tools

  Examples of clients:                 Examples of servers:
  • Claude Desktop                     • File system MCP server
  • Cursor IDE                         • GitHub MCP server
  • Your LangGraph agent               • Database query server
  • Your ADK agent                     • Web search server


============================================================
  SUMMARY — The MCP Lifecycle
============================================================

  1. INITIALIZE    Client and server exchange identities and
                   capabilities (what features each supports).

  2. DISCOVER      Client asks for available tools, resources,
                   and prompts. Gets back schemas it can pass
                   to the LLM for function calling.

  3. USE           Client calls tools/reads resources as needed.
                   Each call is a JSON-RPC request/response pair.

  4. CLOSE         Client closes the connection (not shown here).

  Key insight: The AI agent (LLM) never talks to the MCP server
  directly. The CLIENT application sits in between — it takes the
  LLM's tool-call decisions and translates them into MCP requests.

  ┌─────┐    tool call    ┌────────┐    MCP request    ┌────────┐
  │ LLM │ ──────────────▶ │ Client │ ────────────────▶ │ Server │
  │     │ ◀────────────── │  App   │ ◀──────────────── │        │
  └─────┘    tool result   └────────┘    MCP response   └────────┘

  Done! You've seen the complete MCP message flow.

============================================================
  MCP: PURPOSE AND CORE IDEAS
============================================================

  Before MCP, every AI app needed a custom adapter for
  every tool — like needing a different charger for every
  phone. MCP is the universal charging standard for AI.

============================================================
  THE N×M INTEGRATION PROBLEM
============================================================

  Agents (5): Claude Desktop, Cursor IDE, LangGraph App, ADK Agent, Custom Bot
  Tools  (10): File System, GitHub, Slack, Database, Web Search, Email, Calendar, Jira, AWS S3, Stripe API

  ┌─────────────────────────────────────────────────┐
  │  WITHOUT MCP (custom adapters)                  │
  │                                                 │
  │  Each agent needs a custom integration for      │
  │  each tool. That's 5 × 10 = 50 integrations.     │
  │                                                 │
  │  Agent A ──adapter──▶ Tool 1                    │
  │  Agent A ──adapter──▶ Tool 2                    │
  │  Agent A ──adapter──▶ Tool 3                    │
  │  Agent B ──adapter──▶ Tool 1  (duplicated!)     │
  │  Agent B ──adapter──▶ Tool 2  (duplicated!)     │
  │  ... 50 total adapters to build and maintain   │
  └─────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────┐
  │  WITH MCP (standard protocol)                   │
  │                                                 │
  │  Each agent implements MCP client once.          │
  │  Each tool implements MCP server once.           │
  │  That's 5 + 10 = 15 implementations.             │
  │                                                 │
  │  Agent A ──MCP──┐                               │
  │  Agent B ──MCP──┤     ┌──MCP──▶ Tool 1          │
  │  Agent C ──MCP──┼─────┼──MCP──▶ Tool 2          │
  │  Agent D ──MCP──┤     └──MCP──▶ Tool 3          │
  │  Agent E ──MCP──┘                               │
  │                                                 │
  │  Savings: 50 → 15 = 70% less work!             │
  └─────────────────────────────────────────────────┘

============================================================
  CORE PRIMITIVE 1: Resources (Read-Only Data)
============================================================

  Resources are like files you can read but not modify.
  They let the AI agent gather information safely.

  Resource: Project README
    URI:         file:///project/README.md
    MIME type:   text/markdown
    Description: Main documentation file for the project

  Resource: Active User Count
    URI:         db://users/active-count
    MIME type:   application/json
    Description: Current number of active users in the database

  Resource: London Weather
    URI:         api://weather/london
    MIME type:   application/json
    Description: Current weather conditions in London

============================================================
  CORE PRIMITIVE 2: Tools (Actions)
============================================================

  Tools let the AI agent DO things — they have side effects.
  Each tool defines its inputs via JSON Schema so the LLM
  knows exactly how to call it.

  Tool: send_email
    Description: Send an email to a recipient
    Parameters:
      • to (string, required): Recipient email address
      • subject (string, required): Email subject line
      • body (string, required): Email body text

  Tool: create_github_issue
    Description: Create a new issue in a GitHub repository
    Parameters:
      • repo (string, required): Repository (owner/name)
      • title (string, required): Issue title
      • body (string, required): Issue description
      • labels (array, optional): Labels to apply

  --- Simulated Tool Call ---
  Call:   {
  "tool": "send_email",
  "arguments": {
    "to": "team@example.com",
    "subject": "Weekly Report",
    "body": "Here are this week's highlights..."
  }
}
  Result: Email sent successfully to team@example.com

============================================================
  CORE PRIMITIVE 3: Prompts (Reusable Templates)
============================================================

  Prompts are like fill-in-the-blank templates.
  The server defines them, the client fills in values.

  Prompt: code_review
    Description: Review code for bugs, style, and improvements
    Arguments:
      • code (required): The code to review
      • language (required): Programming language
      • focus (optional): What to focus on

  Prompt: summarize_document
    Description: Create a concise summary of a document
    Arguments:
      • document (required): The text to summarize
      • max_length (optional): Maximum summary length

  --- Filled-In Prompt Example ---
  Template: code_review
  Arguments: {code: 'def add(a,b): return a+b', language: 'python'}
  Result sent to LLM:
    'Review this python code for bugs, style, and improvements:
     def add(a,b): return a+b'

============================================================
  CORE PRIMITIVE 4: Sampling (Server Asks Client's LLM)
============================================================

  Normally: Client calls Server (agent uses a tool)
  Sampling: Server calls Client (tool asks the LLM for help)

  This is like a mechanic (server/tool) asking you (client)
  to call an expert friend (LLM) for a second opinion.

  --- Simulated Sampling Flow ---
  Server → Client: 'I need LLM help to Categorize security findings by severity'
  Messages: [
  {
    "role": "user",
    "content": "I analyzed the codebase and found 3 potential security issues. Please categorize each by severity (low/medium/high):\n1. SQL query uses string concatenation instead of parameterized queries\n2. API key is stored in a comment in the source code\n3. Error messages expose internal file paths"
  }
]

  Client checks: Is this request appropriate? [APPROVED]
  Client → LLM: (forwards the request)
  LLM → Client: 'Issue 1: HIGH, Issue 2: HIGH, Issue 3: MEDIUM'
  Client → Server: (forwards the response)

  Key point: The CLIENT controls access to the LLM.
  The server cannot bypass the client's approval.

============================================================
  TRANSPORT TYPES
============================================================

  Transport = HOW messages travel between client and server.
  The MCP messages themselves are the same regardless of transport.

  stdio (Standard I/O)
    Mechanism: Client spawns server as subprocess, uses stdin/stdout
    Direction: Bidirectional via pipes
    Use case:  Local tools (file system, git, local databases)
    Status:    Stable, widely supported

  SSE (Server-Sent Events)
    Mechanism: HTTP POST for requests, SSE stream for responses
    Direction: Client→Server: POST | Server→Client: SSE stream
    Use case:  Remote servers, web-based tool providers
    Status:    Stable but being superseded by Streamable HTTP

  Streamable HTTP
    Mechanism: Single HTTP endpoint, bidirectional streaming
    Direction: Both directions via single endpoint
    Use case:  Production deployments (recommended for new servers)
    Status:    Newest transport, actively being adopted

  How to choose:
    • Building a local tool? → Use stdio
    • Building a remote service? → Use Streamable HTTP
    • Need backward compat? → Support SSE + Streamable HTTP

============================================================
  CAPABILITY NEGOTIATION
============================================================

  During initialization, client and server exchange
  capabilities. This way each side knows what the other
  supports — no guessing, no trial-and-error.

  Client capabilities:
    • Sampling support: True
    • Filesystem roots: True

  File System Server:
    Supports: Tools, Resources

  Code Analysis Server:
    Supports: Tools, Resources, Prompts, Sampling
    Match: Both support sampling — server can request LLM help

  Weather Server:
    Supports: Tools

  The negotiation ensures graceful degradation:
  if the server offers something the client can't handle,
  that feature is simply not used (no crashes).

============================================================
  PRE-MCP vs POST-MCP WORLD
============================================================

  Aspect                   Pre-MCP                            Post-MCP
  ──────────────────────── ────────────────────────────────── ──────────────────────────────────────

  Adding a new tool
    Before: Write custom adapter code for each AI app that needs it
    After:  Implement one MCP server — all MCP clients can use it instantly

  Supporting a new AI app
    Before: Write custom integrations for each tool it needs
    After:  Implement one MCP client — all MCP servers are accessible

  Tool discovery
    Before: Hardcoded tool definitions in each application
    After:  Dynamic discovery via tools/list — tools self-describe

  Security model
    Before: Each integration has its own auth approach
    After:  Standardized capability negotiation and approval flows

  Transport flexibility
    Before: Each tool decides its own communication method
    After:  Standard transports (stdio, SSE, Streamable HTTP)

  Error handling
    Before: Different error formats per integration
    After:  Consistent JSON-RPC error codes and messages

  Ecosystem growth
    Before: Linear — each new tool requires work from each app
    After:  Network effect — each new server benefits all clients

============================================================
  ALL FOUR PRIMITIVES IN ACTION
============================================================

  Scenario: An AI agent helps a developer review code.
  The agent connects to a 'Code Assistant' MCP server.

  Step [RESOURCE]: Agent reads the source code file
    MCP call: resources/read → file:///src/main.py → returns file contents

  Step [TOOL]: Agent runs the linter on the code
    MCP call: tools/call → run_linter(file='main.py') → returns lint warnings

  Step [PROMPT]: Agent uses the server's review template
    MCP call: prompts/get → code_review(code=..., language='python') → formatted prompt       

  Step [SAMPLING]: Server asks LLM to categorize findings
    MCP call: sampling/createMessage → LLM categorizes bugs by severity → returns categories  

  Result: The agent produces a comprehensive code review
  by combining read-only data (Resource), actions (Tool),
  standardized instructions (Prompt), and AI assistance
  within the tool itself (Sampling).

============================================================
  KEY TAKEAWAYS
============================================================

  1. MCP solves the N×M integration problem by providing a
     standard protocol — build once, connect everywhere.

  2. Four core primitives:
     • Resources — read-only data (safe, no side effects)
     • Tools     — actions (side effects, needs approval)
     • Prompts   — reusable templates (consistent quality)
     • Sampling  — reverse flow (server asks client's LLM)

  3. Three transport types:
     • stdio          — local tools (simple, secure)
     • SSE            — remote servers (being replaced)
     • Streamable HTTP — production (recommended for new work)

  4. Capability negotiation ensures client and server only use
     features both sides support — graceful, no crashes.

  5. MCP creates a network effect: every new server benefits
     ALL existing clients, and vice versa.

  Done! You now understand why MCP exists and how it works.

======================================================================
  SUMMARY: WHAT YOU LEARNED
======================================================================

  1. MCP CLIENT PATTERN: connect → initialize → list_tools → call_tool
     This same pattern works with ANY MCP server, regardless of what
     tools it provides. That's the power of a standard protocol.

  2. STDIO TRANSPORT: The client spawns the server as a subprocess.
     Communication happens over stdin/stdout pipes using JSON-RPC.
     No HTTP servers, no ports, no CORS — just processes and pipes.

  3. LLM-DRIVEN TOOL SELECTION: Give the LLM the tool descriptions
     from list_tools(), and it picks the right tool + arguments.
     This is how you build truly dynamic agents that adapt to whatever
     tools are available.

  4. ERROR HANDLING: Always validate before calling — check that the
     server exists, the tool name is valid, and arguments match the
     schema. In production, add timeouts and circuit breakers.

PART 1: WHY A2A? (The Phone Directory for AI Agents)
======================================================================

Analogy: Before phone books, you had to personally know everyone's
number.  The phone book (A2A) lets you look up any business (agent)
by what they do, call them, and get service — without knowing them
personally.

The A2A protocol has 4 core pieces:
  1. Agent Card    — "Here's who I am and what I can do"
  2. Task          — "Here's what I need you to do"
  3. Message       — "Here's information for the task"
  4. Artifact      — "Here's the finished output"

PART 3: TASK LIFECYCLE (The State Machine)
======================================================================

A Task goes through these states:

  ┌──────────┐     ┌──────────┐     ┌───────────┐
  │ submitted │────▶│ working  │────▶│ completed │
  └──────────┘     └────┬─────┘     └───────────┘
                        │
                        │ (needs more info)
                        ▼
                  ┌──────────────┐
                  │input-required│──── (client provides input) ───▶ working
                  └──────────────┘
                        │
                        │ (or)
                        ▼
                  ┌──────────┐
                  │  failed  │
                  └──────────┘

  • submitted:      Task received, queued for processing
  • working:        Agent is actively working on it
  • input-required: Agent needs more info from the client
  • completed:      Task finished successfully (has artifacts)
  • failed:         Task failed (has error message)
  • canceled:       Task was canceled by the client

PART 5: Agent Discovery (The Phone Book Lookup)
======================================================================

How does Agent A find Agent B?

  1. Agent B hosts its Agent Card at:
     GET https://agent-b.example.com/.well-known/agent.json

  2. Agent A fetches that URL and reads the card.

  3. Agent A checks if Agent B has the skills it needs.

  4. If yes, Agent A sends a task to Agent B's URL.

The ".well-known" convention comes from web standards (RFC 8615).
It's like a standardized address — every agent puts their card
in the same place, so you always know where to look.

PART 1: MCP vs A2A — Feature Comparison
======================================================================

  Feature              MCP                                 A2A
  ──────────────────── ─────────────────────────────────── ───────────────────────────────────
  Purpose              Connect agents to TOOLS             Connect agents to AGENTS
  Analogy              USB port (plug in any device)       Phone network (call any agent)     
  Relationship         Agent ↔ Tool                        Agent ↔ Agent
  Discovery            Server declares tools               Agent Card at /.well-known/        
  Communication        JSON-RPC (stdio/SSE)                JSON-RPC (HTTP)
  Unit of work         Tool call (request/response)        Task (state machine)
  Statefulness         Stateless (each call independent)   Stateful (task lifecycle)
  Who runs it          Tool server (passive)               Agent server (active, has LLM)     
  Authentication       Per-server config                   In Agent Card (OAuth2, API key)    
  Streaming            SSE transport                       Streaming task responses
  Spec owner           Anthropic (open standard)           Google (open standard)
  Key concept          Tools, Resources, Prompts           Agent Cards, Tasks, Artifacts

PART 2: When to Use Which
======================================================================

Use MCP when:
  ✅ You need to call a database, API, or external service
  ✅ The "other side" is a tool (not an AI agent)
  ✅ You want stateless request/response
  ✅ You're adding capabilities to a single agent

Use A2A when:
  ✅ You need two AI agents to collaborate on a task
  ✅ The "other side" has its own LLM and decision-making
  ✅ You need a task lifecycle (submitted → working → completed)
  ✅ You're building a multi-agent system across services

Use BOTH when:
  ✅ Agent A uses MCP to access tools (databases, APIs)
  ✅ Agent A communicates results to Agent B via A2A
  ✅ Agent B uses its own MCP tools to process the results
  ✅ This is the HYBRID pattern — and it's the most powerful!

PART 5: Common Integration Patterns
======================================================================

  📋 Pattern 1: MCP-Only (Single Agent + Tools)
     One agent uses MCP to access multiple tools
     When: Simple tasks that don't need agent collaboration
     Flow: User → Agent → [MCP Tool 1, MCP Tool 2, ...] → User
     Example: A weather bot that checks weather + traffic + news

  📋 Pattern 2: A2A-Only (Agent Delegation)
     Agents delegate tasks to specialists without tools
     When: Pure LLM tasks (writing, analysis, translation)
     Flow: User → Agent A → Agent B → Agent A → User
     Example: Translate + summarize: Translator agent → Summarizer agent

  📋 Pattern 3: Hybrid Sequential
     Agent uses MCP for data, then sends to another agent via A2A
     When: Research-then-write, gather-then-analyze
     Flow: User → Agent A → [MCP tools] → Agent B (via A2A) → User
     Example: Research agent gathers data → Writer agent produces report

  📋 Pattern 4: Hybrid Parallel
     Multiple agents work simultaneously, each with their own MCP tools
     When: Complex tasks that can be parallelized
     Flow: User → Supervisor → [Agent A + MCP, Agent B + MCP] → Supervisor → User
     Example: Parallel research on different sub-topics, then merge

  📋 Pattern 5: Hierarchical Hybrid
     A supervisor coordinates sub-agents, each with MCP tools
     When: Large, multi-step projects
     Flow: User → Planner → [Researcher+MCP, Writer+MCP, Reviewer] → User
     Example: Full content pipeline: plan → research → write → review → publish


======================================================================
PART 6: Decision Matrix — Which Protocol Do I Need?
======================================================================

  Ask yourself these questions:

  Q1: Does the other side have an LLM / make decisions?
      YES → A2A (it's an agent)
      NO  → MCP (it's a tool)

  Q2: Do I need a multi-step conversation?
      YES → A2A (tasks have lifecycles)
      NO  → MCP (single request/response)

  Q3: Does the other side need to be discoverable by anyone?
      YES → A2A (Agent Cards for discovery)
      NO  → MCP (configured per-client)

  Q4: Is the interaction stateless (fire-and-forget)?
      YES → MCP (tool calls are independent)
      NO  → A2A (tasks maintain state)

  Q5: Am I building a single agent or a multi-agent system?
      Single → MCP (add tools to your agent)
      Multi  → A2A + MCP (agents coordinate via A2A, use tools via MCP)

======================================================================
KEY TAKEAWAYS
======================================================================

1. MCP = agent-to-TOOL protocol (like a USB port)
2. A2A = agent-to-AGENT protocol (like a phone network)
3. They're COMPLEMENTARY, not competing — use both!
4. MCP is at the tool layer, A2A is at the coordination layer
5. The hybrid pattern (MCP + A2A) is the most powerful for real systems
6. Decision matrix: Does the other side think? A2A. Is it a tool? MCP.

PART 7: Production Checklist Summary
======================================================================

PRODUCTION READINESS CHECKLIST:

  [1] RETRIES (MCPRetryClient)
      - Exponential backoff: 1s -> 2s -> 4s
      - Max 3 attempts before giving up
      - Connection pooling to reduce latency
      - Only retry on transient errors (connection, timeout)

  [2] COST CONTROL (A2ACostGuard)
      - Per-task budget: prevent any single task from running away
      - Per-session budget: overall spending cap
      - Pre-flight check: verify budget BEFORE making the LLM call
      - Usage logging: know exactly where money goes

  [3] HEALTH CHECKS (HealthChecker)
      - MCP servers: TCP connectivity check
      - A2A agents: HTTP /health endpoint
      - Structured reports for monitoring systems
      - Parallel checks for speed

  [4] DEPLOYMENT (DeploymentConfig + Docker Compose)
      - Type-safe configuration with Pydantic
      - Docker Compose for multi-service orchestration
      - Secrets management (files, not env vars)
      - Health check integration in container orchestration

  [5] GRACEFUL SHUTDOWN (GracefulShutdown)
      - Signal handling (SIGINT, SIGTERM)
      - Task draining with timeout
      - Connection cleanup
      - Works as a context manager (with statement)

KEY PRINCIPLE:
  Each of these is a SEPARATE concern.  You compose them together
  like LEGO bricks.  The MCPRetryClient doesn't know about costs.
  The CostGuard doesn't know about health checks.  Each piece
  does one thing well, and they combine into a robust system.

SERVER 1: DatabaseToolServer (SQLite Read-Only)
======================================================================

  Tools:
    - query_db(sql, db_path) — Execute SELECT queries (read-only)
    - describe_table(table_name, db_path) — Get table schema (columns, types)
    - list_tables(db_path) — List all tables in a database

  Safety: Rejects INSERT, UPDATE, DELETE, DROP, ALTER, CREATE, TRUNCATE

SERVER 2: CodeExecutionServer (Sandboxed Python)
======================================================================

  Tools:
    - run_python(code, timeout_seconds) — Execute Python in a subprocess
    - run_python_safe(code) — Same but with restricted builtins

  Security: Blocks os.system, subprocess, eval, exec, __import__
  Timeout: Default 5 seconds, max 30 seconds


======================================================================
SERVER 3: WebFetchServer (Web Content Fetching)
======================================================================

  Tools:
    - fetch_url(url) — Fetch URL and return stripped text content
    - extract_links(url) — Fetch URL and extract all href links
    - fetch_json(url) — Fetch URL and parse response as JSON

  Note: Uses urllib (stdlib) to avoid extra dependencies.


Key Takeaways:
  1. TOOL SERVERS are specialized workshops — each handles one domain
  2. MCP PROTOCOL is the universal door — same client code for any server
  3. SAFETY FIRST: read-only DB, sandboxed code, validated URLs
  4. COMPOSABILITY: agents mix and match servers for complex workflows
  5. SELF-TESTING: every server should be testable in isolation
  6. GRACEFUL DEGRADATION: handle failures at each layer

SERVER 1: DatabaseToolServer (SQLite Read-Only)
======================================================================

  Tools:
    - query_db(sql, db_path) — Execute SELECT queries (read-only)
    - describe_table(table_name, db_path) — Get table schema (columns, types)
    - list_tables(db_path) — List all tables in a database

  Safety: Rejects INSERT, UPDATE, DELETE, DROP, ALTER, CREATE, TRUNCATE


======================================================================
SERVER 2: CodeExecutionServer (Sandboxed Python)
======================================================================

  Tools:
    - run_python(code, timeout_seconds) — Execute Python in a subprocess
    - run_python_safe(code) — Same but with restricted builtins

  Security: Blocks os.system, subprocess, eval, exec, __import__
  Timeout: Default 5 seconds, max 30 seconds


======================================================================
SERVER 3: WebFetchServer (Web Content Fetching)
======================================================================

  Tools:
    - fetch_url(url) — Fetch URL and return stripped text content
    - extract_links(url) — Fetch URL and extract all href links
    - fetch_json(url) — Fetch URL and parse response as JSON

  Note: Uses urllib (stdlib) to avoid extra dependencies.


Key Takeaways:
  1. TOOL SERVERS are specialized workshops — each handles one domain
  2. MCP PROTOCOL is the universal door — same client code for any server
  3. SAFETY FIRST: read-only DB, sandboxed code, validated URLs
  4. COMPOSABILITY: agents mix and match servers for complex workflows
  5. SELF-TESTING: every server should be testable in isolation
  6. GRACEFUL DEGRADATION: handle failures at each layer

KEY TAKEAWAYS
======================================================================

1. Version your agents with semantic versioning (major.minor.patch)
2. Store prompts in a registry, not hardcoded in source code
3. A/B test prompt changes before rolling them out to all users
4. Extend Agent Cards with version metadata for compatibility checks
5. Keep a structured changelog for every agent version
6. Rollback is your safety net — always keep old versions available

Best practices:
  • MAJOR bump: breaking changes to output format or removed tools
  • MINOR bump: new tools, new skills, improved prompts
  • PATCH bump: bug fixes, typo corrections
  • Always test before promoting a new version
  • Use the registry's usage stats to compare version performance
